## 3. DESIGN DIRECTION

Do **not** default to a generic "cream background + terracotta accent" AI-template look. Design a palette and type system that feels like a warm, well-loved, slightly nostalgic neighborhood bookstore that has survived three decades — think worn leather spines, manga ink, and paper — not a generic SaaS landing page.

**Design system to establish and document in a `DESIGN.md` file before writing code:**
- **Color palette**: 4–6 named hex values. Suggested direction (adapt, don't copy blindly): a deep maroon/oxblood primary (spine-of-an-old-book feel), a warm mustard/ochre secondary (manga cover energy), an aged-paper cream/ivory background (not stark white), charcoal-ink text (not pure black), and one accent for CTAs (e.g. a muted saffron or brick red — festive but not garish, since this is an Indian storefront and festive discounts are a real feature).
- **Typography**: pair a characterful serif or slab-serif for headings (evokes classic book typography) with a clean humanist sans for body text, plus a monospace or condensed utility face for prices/labels/rental countdowns.
- **Layout signature**: one memorable element — e.g. a hand-drawn-style "shelf" divider between categories, a stamped/ticket-style visual for rental due dates, or a card design that mimics a library index card for each book. Pick ONE and execute it with restraint.
- **Motion**: subtle page-load reveal, hover states on book cards, no gratuitous animation. Respect `prefers-reduced-motion`.
- **Mobile-first**: every layout must be designed for a 375px viewport first, then scaled up to tablet (768px) and desktop (1280px+). Test all interactive flows (cart, rental, buy-back form) on mobile breakpoints, not just visually but functionally (tap targets ≥44px, no horizontal scroll, sticky mobile cart/CTA bar).

Before writing code, produce a short design plan (palette, type pairing, layout concept with ASCII wireframe, signature element) and self-critique it against "does this look like every other AI-generated bookstore site" — revise until it doesn't.

## 4. TECH STACK (production-grade, use exactly this unless you have a strong justified reason to deviate — state the reason if you do)

- **Framework**: Next.js 14+ (App Router), TypeScript throughout (strict mode on)
- **Styling**: Tailwind CSS + a small set of custom CSS variables for the design tokens above
- **Component layer**: shadcn/ui as a base, customized to match the design system (do not ship default shadcn look)
- **Database**: PostgreSQL
- **ORM**: Prisma
- **Auth**: NextAuth.js (Auth.js) — email/password + optional Google login for customers; separate protected `/admin` role for the owner
- **State management**: React Server Components + Zustand (or React Context) only for client-side cart/rental state
- **Forms & validation**: React Hook Form + Zod
- **Payments**: Razorpay (India-first payment gateway) for online orders/rental deposits — integrate in test mode with clear `.env.example` placeholders
- **Image handling/storage**: Cloudinary or Vercel Blob for book cover photos and buy-back submission photos
- **Email**: Resend or Nodemailer for order confirmations, rental due-date reminders, and buy-back status updates
- **Deployment target**: Vercel (frontend + API routes) + a managed Postgres instance (Supabase, Neon, or Railway) — document exact deployment steps
- **Analytics**: Vercel Analytics or Plausible (privacy-friendly)
- **Testing**: Vitest/Jest for unit tests on business logic (rental fee calculation, buy-back offer logic), Playwright for at least the core e2e flows (browse → add to cart → checkout; submit buy-back request; rent a book)
- **Linting/formatting**: ESLint + Prettier, pre-commit hook via Husky + lint-staged
- **CI**: a basic GitHub Actions workflow that runs lint, typecheck, and tests on every PR

- **Coupon/Offer** (id, title, description, discountPercent or discountAmount, validFrom, validTo, isActive, bannerImageUrl?)

Write real Zod schemas mirroring these models for all form validation (client + server side — never trust client-only validation).

## 7. KEY BUSINESS LOGIC TO IMPLEMENT (not just UI — actual working logic)

1. **Rental fee calculator**: `dailyRate * numberOfDays`, plus deposit, plus overdue late fee calculation once `dueDate` passes and book isn't returned. Write this as a pure, unit-tested function.
2. **Buy-back workflow**: state machine (Pending → Reviewed → Offer Made → Accepted/Rejected → Completed) with admin-only transitions and customer-visible status tracking.
3. **Festive discount engine**: coupons apply to cart total or specific categories, respect `validFrom`/`validTo`/`isActive`, and the homepage banner automatically shows/hides based on active offers.
4. **Book Wiki search**: full-text search (Postgres `tsvector` or a simple ILIKE search to start) across title/author/synopsis/glossary.
5. **Inventory sync**: when a book is sold or a rental starts, stock/availability updates in real time so two customers can't buy the last copy or rent an already-rented book.

## 8. NON-FUNCTIONAL REQUIREMENTS (production bar — do not skip)

- **Mobile responsiveness**: fully functional and visually polished at 375px, 414px, 768px, 1024px, 1440px. Sticky bottom cart/CTA bar on mobile for product pages.
- **Performance**: Lighthouse score 90+ on mobile for Performance, Accessibility, Best Practices, SEO. Use `next/image` for all book covers, lazy-load below-the-fold content.
- **SEO**: proper metadata per page, OpenGraph tags, sitemap.xml, robots.txt, semantic HTML, and structured data (schema.org `Book` and `LocalBusiness` markup — this is a real local business in Bardoli, so `LocalBusiness` schema with address/phone/hours matters for local search).
- **Accessibility**: WCAG AA — visible keyboard focus states, proper alt text, sufficient color contrast, ARIA labels on interactive components, reduced-motion support.
- **Security**: hash passwords properly (bcrypt/argon2), CSRF protection on forms, rate-limit the contact/buy-back/rental forms, sanitize all user input, never expose Razorpay secret keys client-side, role-gate all `/admin` routes server-side (not just hidden UI).
- **Error handling**: proper 404/500 pages matching the design system, form-level error states with clear plain-language messages, empty states designed with intent (e.g. empty cart, no search results, no active rentals).
- **Environment config**: a complete `.env.example` with every required variable documented (DATABASE_URL, NEXTAUTH_SECRET, RAZORPAY_KEY_ID/SECRET, CLOUDINARY/BLOB credentials, RESEND_API_KEY, etc.)
- **Documentation**: a `README.md` with setup instructions, a `DESIGN.md` with the design system decisions, and inline comments on non-obvious business logic (rental fees, buy-back state machine).
